## 요구사항 정의서

소프트웨어의 품질을 보장하기 위해 작성하는 문서

프로그램 만드는데 드는 모든 비용을 산출한다고 생각하면 된다.

다음 내용을 작성해야한다.

ECR (External Communication Requirement)\
외부 통신 요구사항으로, 시스템이 다른 시스템과 통신할 때의 요구사항을 의미합니다. 예를 들어 API나 데이터 전송 규격 등이 여기에 해당됩니다.

SFR (Software Functional Requirement)\
소프트웨어 기능 요구사항으로, 소프트웨어가 수행해야 하는 기능적 요구사항을 명시합니다. 예를 들어 사용자 로그인, 데이터 처리 기능 등이 포함됩니다.

PER (Performance Requirement)\
성능 요구사항으로, 시스템의 성능과 관련된 요구사항을 의미합니다. 응답 시간, 처리량, 대기 시간 등 성능과 관련된 요구가 여기에 포함됩니다.

SIR (System Interface Requirement)\
시스템 인터페이스 요구사항으로, 다른 시스템과의 인터페이스에 대한 요구사항을 의미합니다. 프로토콜, 데이터 포맷, 연결 방식 등 시스템 간의 통신에 필요한 사항이 포함됩니다.

DAR (Data Requirement)\
데이터 요구사항으로, 시스템이 처리하거나 저장해야 하는 데이터와 관련된 요구사항입니다. 데이터의 구조, 저장 방식, 형식 등이 포함됩니다.

TER (Test Requirement)\
테스트 요구사항으로, 시스템이 요구사항을 만족하는지 검증하기 위한 테스트에 대한 요구사항입니다. 테스트의 종류, 조건, 결과 검증 방법 등이 포함됩니다.

SER (Safety Requirement)\
안전 요구사항으로, 시스템의 안전성과 관련된 요구사항을 의미합니다. 시스템 사용 시 위험 요소를 방지하기 위한 안전 조치 등이 포함됩니다.

QUR (Quality Requirement)\
품질 요구사항으로, 시스템의 품질과 관련된 요구사항을 의미합니다. 신뢰성, 유지보수성, 이식성 등 소프트웨어 품질과 관련된 사항이 포함됩니다.

COR (Compliance Requirement)\
준수 요구사항으로, 법적, 규제적, 산업 표준 등에 맞춰 시스템이 준수해야 하는 요구사항입니다.

PMR (Privacy Requirement)\
개인정보 요구사항으로, 사용자 개인정보 보호와 관련된 요구사항을 의미합니다. 데이터 암호화, 접근 제어, 개인정보 보호법 준수 등이 포함됩니다.

PSR (Product Security Requirement)\
제품 보안 요구사항으로, 시스템의 보안을 보장하기 위한 요구사항입니다. 보안 인증, 접근 제어, 데이터 보호 등을 포함합니다.

## 기능 리스트

요구 사항 리스트에서 기능을 추출하여 정리한 것\
예를들어 `SPR-0001`이라는 요구 사항이 있으면 기능 리스트는 `SPR-0001`에 대해 어떻게 구현할 것인지 작성할 수 있다.

| 요구사항 ID | 요구 내용 | 기능 ID | 기능명 | 기능내용 | 업무구분 |
| --- | --- | --- | --- | --- | --- |
| SPR-0001 | 로그인 | F-0001 | 로그인 | 로그인 기능을 구현한다. | 사용자 |

## 엔티티 정의서

시스템에서 사용하게 될 엔티티를 정의한다.\
테이블을 미리 정의한도고 볼 수 있다.\
예를들어 사용자 엔티티를 정의할 수 있다.

| 주제 | 약어 | 논리 테이블명 | 물리 테이블명 | 설명 | 비고 |
| --- | --- | --- | --- | --- | --- |
| 사용자 | USR | 사용자 | TB_USR | 사용자 정보를 저장한다. |  |

## 프로그램 목록서

시스템에서 사용할 프로그램 목록을 정의한다.\
웹 개발 기준 어떤 화면을 개발할 것인지 페이지 라우터 단위로 정의하면 된다.

예를들어 사용자 목록 화면을 개발할 것이라면 `사용자 목록`이라는 프로그램을 개발할 것이라고 정의할 수 있다.

| 프로그램 ID | 프로그램명 | 프로그램 설명 | 업무구분 |
| --- | --- | --- | --- |
| P-0001 | 사용자 목록 | 사용자 목록을 조회한다. | 사용자 |

- 업무구분
- 화면ID
- 화면명

## WBS

Work Breakdown Structure\
업무 분할 구조로 진행할 업무를 관리하는 방법론이다.\
보통 Notion, atlazian Jira, readmine, trello, excel 등 을 상용하여 작성한다.

일종의 Todo list로 생각할 수 있다.

git projects로 관리하기도 하는데, 솔루션이나 서비스 업체에서 간혹 사용하는 것 같다.


## SOA 단계

착수 -> 설계 -> 개발/구축 -> 테스트 -> 운영/유지보수 단계를 거친다.

설계시 요구사항 정의서, 엔티티 정의서, 프로그램 목록서, WBS 등을 작성한다.

착수 단계에서의 요구 사항을 바탕으로 RFP를 작성하고, RFP를 바탕으로 제안서를 작성한다.


## 주간 업무 보고

DA, AA, 기획, 공통, 등 각 파트에서 주간 업무 보고를 작성한다.

파트마다 업무를 취합하는 방식이 다르다.

action-item 시트를 만들어서 공유하거나 (일종의 todo list)

commit log, PR, kanban board 등을 사용하여 관리한다.

## 업무 매뉴얼 (가이드 문서)

클라이언트가 시스템을 사용하는데 도움을 줄 수 있는 문서를 작성한다.

일종의 사용자 매뉴얼이라고 생각하면 된다.

예를들어 `관리자 ~파트 사용 매뉴얼`,  `일반 사용자 접수 매뉴얼`, `개발자 가이드` 등이 포함된다.


## 화면 정의서

개발자가 화면을 개발할 때 참고하는 문서로 디자인, 개발, 테스트 등의 업무를 수행할 때 활용한다.

보통 API 주소와 어떤 데이터가 필요한지, 어떤 데이터를 보여줘야 하는지 등을 작성한다.

디자인 가이드는 보통 피그마로 제공하고, url만 화면 정의서에 작성한다.
