# 그림으로 이해하는 시스템 설계

다음 순서로 시스템을 설계한다.

```txt
1. 시스템화 요구사항
	1-1. 다양한 상황과 요인
2. 시스템 아키텍처 설계 (전체 설계)
	2-1. 신뢰성/안정성 종합 설계
3. 설계 규칙 및 방침 설계 (전체 설계)
	3-1. 환경 설계
	3-2. 성능 설계
	3-3. 보안 설계
	3-4. 운영 방식 설계
	3-5. 외부 접속
	3-6. 표준화
4. 개별 설계
	4-1. 입출력 설계
	4-2. DB 설계
	4-3. 로직 설계
	4-4. 네트워크 설계
	4-5. 서버 설계
```
## 설계서의 종류

설계서의 종류를 크게 세가지로 나눌 수 있다.\
공식적인 용어는 아닌 것 같다.

- 관리계
- 전체파악계
- 개별계

**관리계**\
요구사항 명세서, WBS, IA, 화목 목록 정의서, 테이블 목록, 프로그램 목록 등\
엑셀이나 테이블로 정리되어 있으면 보통 관리계이다.

**전체파악계**\
포폴만들 떄 프로젝트마다 꼭 시스템 아키택쳐를 넣었던 기억이 있는데\
이런게 전처파악계이다.\
전체 배치설계, 네트워크 전체 구성도 등 보통 아이콘이 가득한 그림으로 되어 있다.

**개별계**\
위 두 개를 제외한 모든 것이다.\
로직, 레이아웃이이에 해당된다.\
화면 정의서, UML, ERD도 여기에 해당된다.

그림으로 이해하는 시스템 설계 44p 참고


## 1. 시스템화 요구 사항

요구 사항은 다음으로 나뉜다.
- 기능 요구사항
- 비기능 요구사항

**기능 요구사항**은 "영수증 PDF 다운로드", "신용카드 결제 기능" 같이 시스템에 필요한 기능을 말한다.\
**비기능 요구사항**은 기능을 제외한 모든 것이다. 주로 인프라에 영향을 끼친다.\
"운영, 우지보수", "시스템 성능", "가용성", "보안" 등의 키워드가 이에 해당된다.

지금까지 본 바로는 보통 다음 방식으로 요구사항이 내려온다.
- 제안 요청서, RFP(Request For Proposal)
- 사전 정보 요청서, RFI(Reqeust For Information)
- 제안 견적 요청서, RFQ(Request For Quotation)

발주사에서 위 요청서를 주면 프로젝트 담당자는 **요구사항 정의서**를 만든다.\
발주사에게 제공하는 산출물(결과물)에 요구사항 정의서가 들어간다.

요구사항 정의서는 **관리계**로 다음 항목을 정리해서 작성한다.
- 시스템 장비 구성, ECR(Equipment Composition Requirement)
- 기능, SFR(System Function Requirement)
- 성능, PER(Performance Requirement)
- 인터페이스, SIR(System Interface Reqruiement)
- 데이터, DIR(Data Requirement)
- 테스트, TER(Test Requirement)
- 보안, SER(Seciriry Requirement)
- 품질, QUR(Quality Requirement)
- 제약사항, COR(Constraints Requirement)
- 프로젝트관리, PMR(Project Mangement Requirement)
- 프로젝트지원, PSR(Project Support Requirement)
- 컨설팅, CNR(Consulting Reqruiement)
- 유지관리수행, MPR(Maintance Project Requirement)
- 인력관리 ,HMR(Maintance Human Requirement)
- 공사, ENR(Enginerring Requirement)

항목별 시트를 만들어서 작성한다.

참고: [블로그](https://blog.naver.com/f0kerface/221256744258), [공공SW사업-제안요청서-작성을-위한-요구사항-가이드](https://www.cisp.or.kr/wp-content/uploads/2021/03/2.%EA%B3%B5%EA%B3%B5SW%EC%82%AC%EC%97%85-%EC%A0%9C%EC%95%88%EC%9A%94%EC%B2%AD%EC%84%9C-%EC%9E%91%EC%84%B1%EC%9D%84-%EC%9C%84%ED%95%9C-%EC%9A%94%EA%B5%AC%EC%82%AC%ED%95%AD-%EA%B0%80%EC%9D%B4%EB%93%9C-20210219.pdf)


## 2. 시스템 아키텍처 설계

전체 설계에 대해서는 아직 잘 모르겠다.\
요구사항 정의서를 바탕으로 설계를 진행할 것로 보인다.\
BPR/ISP 컨설팅을 받을 수 있을 것 같다.

 BPR(Business Process Re-engineering), 정보처리 엄무 과정을 경영과 엮어서 프로세스를 다시 만들어줌\
ISP(Information Strategy Planning), 정보시스템 계획 작성을 도와줌


참고: [casit BPR/ISP 컨설팅](http://casit.co.kr/173)

## 3. 설계 규칙 및 방침 설계

이것도 잘 모르겠다.\
컨벤션이나 시큐어 코딩을 위한 가이드를 공공기관에서 제공하고 있고,\
회바회 프바프로 보인다.

## 4. 개별 설계

병렬적으로 설계되고, 대부분 산출물(결과물)에 포함된다.\
프로젝트마다 사용하는 설계서가 다르지만 용어는 동일하다.

### 4-1 입출력 설계

- 화면계
- 장표계
- if계
- 기타

#### 화면계
화면 목록 (IA, 요구사항 정의서 등 참조)\
화면 정의서 (스토리보드, 유저 플로우 등에 해당)

화면 공통계 (어플리 케이션 단, 유동적으로 일정 중 변경 가능 감안)\
화면 레이아웃계 (설계 방침, 컨벤션, 디자인 시스템에서 영향을 받음\
uxui 정의서를 참고해수 설계)\
화면 입력 설계 (form 양식에 해당. 화면 정의서와 별계로 입력계 정의서를 만드는게 베스트)

#### 장표계
리포트 툴을 사용하거니, 테이블, pdf 형태의 출력이 주를 이룸,\
이미지 형태의 지류도 있을 수 있기에 데이터가 짤려서 출력되지 않도록 주의.\
장표계는 외부 접속이 필요할 수 있음. if 외부 접속계 참고.

#### if계
외부 접속 대상 목록(타사 서비스를 사용하는경우)

외부 접속 방식 설계(서비스를 제공하는 경우)\
접속 시간 제약, 프로토콜 등 정할 부분이 많음. 보통 배치프로그램이서 많이 사용)\
타사애서 쓰는 것이기에 내부 사정늘 몰라도 쓸 수 있도록 설계 한다.

#### 기타
이메일 설계\
이메일 양식.

추가로 입출력 설계는 이용자의 입장을 고려해야함\
접근성, 편의 기능, ux가 그러함

상태 전이 설계= 배치 프루그램

설계 이해가 목적일 경우 액티비티 다이어그램\
기능에 대한 이해가 목적일 경우 플로우 차트 시퀀스 다이어그램

### 4-2. DB 설계

- 전체 설계
	- 볼륨 목록, 관리계
- RDB
	- db 설정, 개별계
	- 테이블 목록, 관리계
	- 인덱스 목록, 관리계
	- ER diagram, 개별계
	- 테이블 레이아웃, 개별계
- NoSQL
	- db 설정, 개별계
	- 데이터 목록, 관리계
	- 데이터 레이아웃, 개별계
- 파일
	- 파일 목록, 관리계
	- 파일 설계, 개별계

#### 볼륨 목록
Volumn은 용량을 말하는데, 파일의 용량(byte)를 여기서는 말한다.\
예측값을 작성한다.

#### DB 설정
요구 사항을 만족하도록 설정해야하고, DB의 종류에 따라 작동 방식, 제공 기능이 다르다.\
비기능 요구 사항이 큰 영향을 끼치기 때문에 DB 설정에는 전문가가 필요하다.

#### 테이블, 인텍스 목록
인덱스, 테이블 목록을 관리한다.\
이건 잘 안 쓸 것 같고 실제 어플리케이션과 싱크를 마주기 힘들 것 같다.

#### ER 다이어그램
ERD는 개별계로 들어간다.\
실제 개발에서도 많이 참고한다.

#### 테이블 레이아웃
ERD를 작성하면 각 테이블마다 세부적인 정의를 설게한다.\
보통 ERD를 작성하는 도구에 같이 들어가있다.\
항목명, 자료형, 자리, not null, pk, fk, index 여부 등을 작성한다.

#### 파일 설계
일단 텍스트 파일로, json, csv, xml, yaml도 여기에 해당된다.\
로그 파일이나 설정 정보 파일 설계가 여기에 해당한다.\
계별계이지만 보통 엑셀 형태로 관리한다.
### 4-3 로직 설계
보통 UML을 사용한다.
- 기능 목록, 관리계
- 프로그램 목록, 관리계
- 클래스 목록, 관리계
- useacse diagram, 개별계
- activity diagram, 개별계
- class diagram, 전체파악계
- package diagram, 전체파악계
- sequence diagram, 개별계
- process flow diagram (flow chart), 개별계
- data flow diagram (DFD), 개별계
- 상태 전이 설계, 개별계
- 배치 전체 설계, 전체파악계
- 배치 프로그램 목록, 관리계
- 처리 설계(프로그램 사양서), 개별계
- 로그 목록, 관리계
- 오류 코드 목록, 관리계

#### usecase diagram
acter와 시스템의 상호 작용을 설계한다.\
시스템의 처리 부분은 블랙 박스로 둔다.\
화면설계서 혹은 스토리보드로 충분한 경우가 대부분이라 굳이 유스케이스까지 작성하는 경우는 많이 없다.\
사용자가 직감적으로 이해하기 어려운 서비스라면 개발자도 유스케이스가 필요할 수 있다.

#### 액티비티 다이어그램
등장 인물을 정하고, 각각의 처리가 흘러가는 흐름 설계한다.\
요구사항에 만족하는지, 최적의 형태인지를 확인하기 위해 사용한다.

#### 클래스 다이어그램
OOP 프로그래밍에 사용하는 전체파악계이다.\
시스템 아키텍처 설계의 표준화에 사용한다.\
패키지 구조를 설계할 때도 필요할 것으로 보인다.

#### 시퀀스다이어그램
객체 간의 메시지 상호작용을 시간순으로 작성한다.\
복잡하게 작성하려면 끝이 없기때문에 포괄적으로 다루고\
자세한 부분은 플로우차트나, 프로그램 사양서 등을 통해서 처리를 설계한다.

#### process flow diagram, flow chart
실제 소스코드에 가까운 다이어그램이다.\
predo code를 작성한다는 느낌으로 작성한다.

#### 상태 전이 설계
배치처리를 설계할때 사용한다.\
애플리케이션 마다 시간별로 시작 점과 끝을 작성한다.
### 4-4 네트워크 설계
신뢰성/안정성 설계, 성능 설계, 보안 설계를 바탕으로 네트워크를 설계한다.

### 4-5 서버 설계
